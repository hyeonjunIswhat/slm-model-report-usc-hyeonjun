01모델 선택 이유

42dot_LLM-SFT-1.3B는 현대자동차 계열 AI 스타트업 42dot이 한국어·영어 혼합 코퍼스로 처음부터(from scratch) 사전학습한 뒤 SFT(Supervised Fine-Tuning)를 적용한 국내 자체 개발 SLM이다. 선택 이유는 크게 세 가지다. 첫째, 데이터 주권(Data Sovereignty) 관점에서 의미 있는 사례다. 해외 베이스 모델에 한국어 파인튜닝을 얹은 대부분의 모델과 달리, 42dot LLM-PLM은 Jikji 프로젝트, mC4-ko, LBox Open(법률 판결문), KLUE 등 국내 공개 데이터를 포함해 자체 인프라에서 사전학습되었다. 공공기관이 AI를 도입할 때 학습 데이터의 출처와 통제 가능성은 핵심 요건이며, 이 모델은 그 논의를 실질적으로 시작할 수 있는 사례다. 둘째, 1.3B 규모는 온프레미스(On-Premises) 배포 가능성의 최소 기준에 해당한다. 행정기관은 개인정보보호법·전자정부법 등으로 인해 클라우드 외부 전송이 제한되는 경우가 많다. 따라서, F32 텐서 기준 약 5GB 수준의 모델은 단일 GPU 서버 또는 엣지 디바이스에서도 추론이 가능하며, 망분리 환경에서의 로컬 배포 시나리오를 현실적으로 검토할 수 있다. 셋째, CC BY-NC 4.0 라이선스라는 제약이 오히려 분석 포인트가 된다. 비상업적 사용만 허용되므로, 공공기관이 이 모델을 민원 응대나 내부 행정 자동화에 도입할 경우 라이선스 해석과 법적 검토가 선행되어야 한다. 이는 단순 기술 도입을 넘어 조달·법무·정보화 부서의 협업이 필요한 거버넌스 문제이기도 하다.

03학습 데이터와 튜닝 방식

사전학습 데이터 (PLM 단계)
베이스 모델인 42dot_LLM-PLM-1.3B는 한국어·영어 혼합 코퍼스로 사전학습되었다. 공개된 구성은 다음과 같다.

· 한국어: 직지(Jikji) 프로젝트, mC4-ko, LBox Open(법원 판결문·법률 데이터), KLUE(한국어 이해 벤치마크 데이터), 한국어 위키피디아 등
· 영어: The Pile, RedPajama, C4 등 표준 영어 사전학습 코퍼스

LBox Open의 포함은 주목할 만하다. 법원 판결문과 같은 공공 문서는 행정 문서와 문체·형식 면에서 유사성이 높아, 공문서 처리나 법령 기반 질의응답 태스크에 일정한 적합성을 기대할 수 있다.

사전학습에는 NVIDIA A100 80G 256장을 사용하여 약 8일(49,152 GPU 시간)이 소요되었다.

파인튜닝 방식 (SFT 단계)
PLM에 Supervised Fine-Tuning(SFT)을 적용하여 지시 수행(Instruction Following) 능력을 부여했다. SFT 학습 데이터는 42dot이 직접 구축한 Single/Multi-turn 대화 형식이며, 구체적인 데이터 구성 및 규모는 비공개다. 학습에는 A100 GPU로 약 112 GPU 시간이 소요되었다.

RLHF, DPO, LoRA, QLoRA 등의 추가 정렬(Alignment) 기법 적용 여부는 공개 모델 카드 범위 내에서 확인되지 않는다. SFT 이후 추가적인 선호도 학습 단계가 없다는 점은, 지시 추종 능력은 갖추되 안전성·유해성 필터링 측면에서 상용 모델 대비 취약할 수 있음을 시사한다.

04파라미터 및 구조적 특징

ArchitectureLLaMA 2 유사 Transformer Decoder (Causal LM)
Context Length4,096 토큰 — 장문 행정 문서(수십 페이지) 단일 입력 처리에 제약 있음. RAG 파이프라인 구성 시 청크 분할 전략 필수.
Hidden Size2,048
Layers24 레이어
Attention Heads32 Attention Heads / FFN Size: 5,632
TokenizerByte-level BPE — 한국어·영어 각 1,000만 문서 샘플로 어휘 자체 학습. 한국어 전용 vocab 구성으로 토큰 효율 최적화.

05모델 파일 구성

42dot/42dot_LLM-SFT-1.3B 저장소의 주요 파일 구성과 역할은 다음과 같다.

· config.json — 모델 아키텍처 정의 파일. hidden_size(2048), num_hidden_layers(24), num_attention_heads(32), max_position_embeddings(4096) 등 구조적 하이퍼파라미터가 기록되어 있다. 모델을 로드하는 라이브러리가 이 파일을 기준으로 네트워크를 재구성한다.

· model.safetensors — 학습된 가중치 파일. F32(단정밀도 부동소수점) 텐서 포맷으로 저장되어 있으며, 모델 전체 크기는 약 5GB 수준이다. safetensors 포맷은 기존 pickle 기반 .bin 대비 보안성과 로딩 속도가 개선된 표준 포맷이다.

· tokenizer.json / tokenizer_config.json — Byte-level BPE 토크나이저 정의. 어휘(vocab)와 병합 규칙(merges)이 포함되어 있으며, 한국어와 영어를 모두 효율적으로 처리할 수 있도록 자체 학습된 어휘를 사용한다.

· special_tokens_map.json / added_tokens.json — <human>, <bot> 등 대화 구조를 정의하는 특수 토큰 매핑. SFT 과정에서 추가된 토큰이 포함되어 있으며, 추론 시 프롬프트 포맷을 이 구조에 맞춰 구성해야 한다.

· generation_config.json — temperature, top_p, top_k, repeat_penalty 등 추론 기본값 설정 파일. 42dot은 공식 문서에서 temp=0.5, top_p=0.95, top_k=20, repeat_penalty=1.2를 권장한다.

06활용 가능 업무

시나리오 1 — 내부 문서 요약 보조 (망분리 환경)
행정기관은 외부 클라우드 API 활용이 제한되는 경우가 많다. 42dot_LLM-SFT-1.3B는 F32 기준 약 5GB로, 단일 GPU 서버(A100 40GB 또는 RTX 3090 이상)에서 로컬 구동이 가능하다. 보고서·지침·회의록 등 내부 문서를 요약하거나 핵심 항목을 추출하는 보조 도구로 활용 가능하다.

시나리오 2 — RAG 기반 법령·지침 질의응답
4,096 토큰의 컨텍스트 한계로 장문 법령 전체를 단번에 처리하기 어렵지만, 벡터 DB(예: Chroma, Weaviate)와 결합한 RAG(Retrieval-Augmented Generation) 파이프라인을 구성하면 관련 조항 검색 후 답변 생성이 가능하다. 행정규칙·시행령 등을 대상으로 한 내부 QA 시스템 구축 시 적용을 검토할 수 있다.

시나리오 3 — 민원 유형 분류 및 초안 응답 생성
반복적으로 접수되는 민원 텍스트를 유형별로 분류하고, 담당자 검토 전 초안 답변을 자동 생성하는 Pre-draft 보조 도구로 활용 가능하다. SFT 모델이므로 지시(instruction) 형식의 프롬프트에 대한 응답 품질이 Base 모델 대비 우수하다.

단, 모든 시나리오에서 Human-in-the-loop(인간 검토) 구조는 필수다. 1.3B 규모 모델은 할루시네이션 발생 가능성이 대형 모델 대비 높으며, 법령 해석이나 민원 답변은 오류 시 행정 책임 문제로 이어질 수 있다.

07한계와 주의사항

1. 라이선스 제약 — CC BY-NC 4.0
비상업적 사용만 허용된다. 공공기관이 민원 처리 자동화, 유료 서비스 연계, 또는 대국민 서비스에 활용할 경우 "상업적 사용"에 해당하는지 법무 검토가 선행되어야 한다. 공공 조달 측면에서도 이 라이선스 조건은 구매·운영 계약 작성 시 명확한 조항이 필요하다.

2. SFT 데이터 비공개 — 투명성 한계
PLM 단계의 학습 데이터는 일부 공개되었으나, SFT 파인튜닝에 사용된 대화 데이터는 전혀 공개되지 않았다. 데이터사이언스 관점에서 모델의 지시 수행 편향(Instruction Bias), 특정 응답 스타일 고착 등을 사전에 파악하기 어렵다. 공공 도입 시 "학습 데이터 출처 및 구성 공개" 요건을 계약 조건으로 명시해야 한다.

3. 컨텍스트 길이 4,096 토큰
실제 행정 문서(조례, 지침, 감사 보고서 등)는 수만 토큰에 달하는 경우가 많다. 단일 입력 처리 불가로 RAG 또는 청크 분할 전략이 필수적이며, 분할 경계에서의 맥락 단절 문제가 답변 품질 저하로 이어질 수 있다.

4. 안전성·정렬(Alignment) 부재
RLHF·DPO 등 추가 정렬 학습이 적용되지 않아, 유해 콘텐츠 생성이나 편향된 응답에 대한 방어 장치가 미흡할 수 있다. 공공기관 대국민 서비스에 직접 노출되는 용도로는 추가적인 안전성 평가와 필터링 레이어 구축이 반드시 필요하다.

5. 모델 갱신 중단 — 2024년 2월 이후 업데이트 없음
HuggingFace 저장소 기준 마지막 업데이트가 2024년 초다. 이후 한국어 LLM 생태계는 EXAONE 3.5, Qwen 2.5, Kanana 등 더 강력한 모델들이 등장했다. 장기 운영 관점에서 모델 유지보수 및 버전 업데이트 계획 부재는 리스크다.

08종합 의견

42dot_LLM-SFT-1.3B는 "국내 자체 사전학습 + 한국어 최적화 토크나이저 + 온프레미스 배포 가능 규모"라는 조합에서 현재까지도 유효한 레퍼런스 모델이다. 그러나 이 모델의 가장 큰 가치는 현재의 성능보다, 한국 공공 AI 도입 논의에서 반드시 다루어야 할 구조적 문제들을 명확하게 드러낸다는 점에 있다.

라이선스는 기술보다 먼저 해결해야 할 문제다. CC BY-NC 4.0은 연구·내부 실험 목적에는 충분하지만, 실제 공공서비스에 적용하려는 순간 법적 불확실성이 발생한다. 국내 공공기관이 오픈소스 LLM을 도입할 때 Apache-2.0이나 MIT 같은 허용적 라이선스를 우선 요건으로 설정해야 하는 이유가 여기 있다.

SFT 데이터 비공개는 행정 신뢰성 요건과 충돌한다. 공공 AI 시스템에는 '왜 이런 답변이 나왔는가'에 대한 설명 가능성(Explainability)이 요구된다. 학습 데이터 구성조차 공개되지 않은 모델을 행정 결정 보조에 사용하는 것은 책임 소재 측면에서 위험하다. 향후 도입 모델에는 학습 데이터 공개 수준을 조달 평가 항목으로 포함시켜야 한다.

POC(개념 검증) 방향 제안: 이 모델을 행정 도메인 특화 파인튜닝의 베이스라인으로 삼아, 공개된 국민신문고 민원 데이터나 행정안전부 공개 FAQ 데이터를 활용한 LoRA 파인튜닝 실험을 설계하는 것이 현실적인 다음 단계다. 동시에 동일 조건에서 EXAONE-3.5-2.4B 등과 벤치마크 비교를 통해 모델 선택의 정량적 근거를 마련해야 한다.
